# IFRS 16 管理系统需求更新清单

## 1. 文档目的

本文件用于记录相对于既有需求文档的新增要求和口径调整，供 IT 团队增量阅读和评估。

本文件不替代原始需求文档，主要用于：

- 快速识别新增范围
- 统一最新设计口径
- 减少 IT 团队重复阅读主文档的成本

## 2. 本次更新摘要

本次更新主要包括以下几类内容：

- 新增 AI Agent 聊天窗口及文件智能入库要求
- 明确 MVP 技术架构和系统边界
- 明确 OCR、对象存储、部署和基础设施选型
- 更新摊销和查询口径为"底层按天，输出可多粒度"
- 增加自定义财年和自定义期间支持要求

## 3. 需求更新明细

### 3.1 新增 AI Agent 聊天窗口

系统新增 AI Agent 聊天窗口要求，作为合同和租赁信息录入、识别、辅助查询入口。

#### 更新内容

- 用户可通过聊天窗口上传租赁相关文件
- 支持文件类型：
  - Excel
  - PDF
  - scan copy
  - 合同正文
  - 补充协议
  - 租金清单
  - landlord notice
- AI 需具备文件识别、OCR、字段抽取、分类、条款理解能力
- AI 识别结果需根据不同场景进入不同链路

#### 主要处理链路

- 新合同识别并生成合同草稿
- 付款计划识别并生成付款计划草稿
- 合同变更识别并生成事件草稿
- 扫描件 OCR 补录
- 只读查询问答

#### 控制要求

- AI 不得直接写入正式台账
- 必须先进入 AI 草稿层
- 低置信度字段必须人工确认
- 必须保留原文定位、置信度、审核差异、审批记录

### 3.2 新增 MVP 技术架构口径

系统增加了一版明确的 MVP 技术架构方案，目标是先打通核心端到端链路。

#### MVP 技术栈

- 前端：Next.js + TypeScript + Ant Design
- 核心后端：Golang + Gin
- 数据访问：pgx + sqlc + goose
- 数据库：PostgreSQL
- AI / 文档解析服务：Python + FastAPI
- OCR / 文档结构化：PaddleOCR
- 大模型：OpenAI API
- 对象存储：MinIO
- 认证授权：Go 自建 JWT + 基础 RBAC
- 监控排障：slog + pprof + 基础 metrics
- 部署：Docker

#### MVP 核心范围

- 合同上传
- AI 识别
- 合同草稿和付款计划草稿
- 人工确认
- 正式入库
- 基础 IFRS 16 计量
- 基础台账查询
- 基础审计日志

#### MVP 暂缓范围

- Keycloak
- Temporal
- Camunda 8
- OpenTelemetry 全链路
- Prometheus + Grafana 完整监控
- 复杂 lease modification / reassessment 自动化全覆盖
- 披露全覆盖

### 3.3 技术选型口径更新

以下技术选型已明确：

- OCR / 文档结构化：PaddleOCR
- 对象存储：MinIO
- 部署方式：Docker

以下重型基础设施暂不纳入 MVP 默认范围：

- Keycloak
- Temporal
- Camunda 8
- OpenTelemetry + Prometheus + Grafana 全套监控体系

说明：

- 以上组件不是否定，而是出于 MVP 快速落地考虑暂缓
- 后续如审批流、运维复杂度、权限复杂度提升，可在 Phase 2 或 Phase 3 引入

### 3.4 摊销逻辑口径更新

原先需求中提到系统应支持日、月两个层级的计量逻辑。现进一步明确如下：

#### 最新口径

- 系统底层计量引擎采用日级逻辑
- 摊销明细表支持按天输出
- 同一套日级明细可自动汇总成月、季、半年、年报表
- 审计和财务查询可下钻到单日明细

#### 原因

- 零售门店常见月中起租、月中终止、免租期跨月、变更生效日非月末
- 财务需支持季报、半年报、年报和审计抽查
- 日级底层更利于报告截止日精确出数和解释

### 3.5 查询与输出粒度更新

除按天输出外，系统还需要支持由用户自行选择查询和导出粒度。

#### 最新要求

系统应支持以下粒度：

- daily
- monthly
- quarterly
- semi_annual
- annual
- custom

#### 设计要求

- 底层按天计算
- 查询层按用户选择汇总
- 输出层支持按选择粒度导出
- 前端不应默认直接铺开全部日表，应以汇总视图为主，支持 drill down

#### 建议参数

- aggregation_level
- reporting_cutoff_date

### 3.6 新增自定义财年和自定义期间支持

系统新增财年和报告期间灵活配置要求。

#### 最新要求

- 不应默认自然年作为唯一财报口径
- 支持不同企业的自定义年结日
- 支持任意查询和报表期间

#### 需支持的场景

- 12月31日年结
- 3月31日年结
- 2月28日 / 2月29日年结
- 其他客户自定义年结日
- 任意起止日期区间查询和导出

#### 建议参数

- fiscal_year_end_date
- report_start_date
- report_end_date
- reporting_calendar_type

建议 `reporting_calendar_type` 支持：

- calendar_year
- fiscal_year
- custom_range

### 3.7 报表和审计支持口径补充

基于以上调整，系统报表和导出层需要同步满足以下要求：

- 支持日级摊销明细导出
- 支持按月、季、半年、年、自定义区间汇总导出
- 支持按合同、门店、法人、品牌、区域筛选
- 支持报告截止日口径的期末余额输出
- 支持审计按指定生效日回溯

## 4. 对 IT 的直接影响

### 4.1 数据模型影响

需要新增或强化以下数据和参数设计：

- AI 任务表
- AI 草稿表
- OCR 和字段置信度字段
- 原文定位字段
- aggregation_level
- fiscal_year_end_date
- report_start_date
- report_end_date
- reporting_calendar_type

### 4.2 计量引擎影响

- 不建议仅做月级引擎
- 建议从底层就按天设计
- 汇总层由日表自动生成

### 4.3 前端影响

- 需要新增 AI Agent 聊天窗口
- 需要新增合同草稿确认页
- 需要新增付款计划草稿确认页
- 需要新增查询粒度切换能力
- 需要新增按自定义期间查询和导出能力

### 4.4 接口影响

需要增加或补强以下接口：

- 文件上传接口
- OCR / AI 识别接口
- 草稿保存接口
- 草稿确认入库接口
- 自定义报表查询接口
- 自定义区间导出接口

## 5. 建议 IT 下一步动作

建议 IT 团队按以下顺序处理本次更新：

1. 先确认 AI Agent 是否纳入 MVP 第一阶段
2. 确认底层计量引擎是否按日级设计
3. 确认查询和导出层是否支持多粒度和自定义期间
4. 更新数据库字段和 API 设计
5. 更新前端页面和工作流范围

## 6. 结论

本次更新的核心变化，不是简单新增几个字段，而是明确了系统需要具备以下能力：

- 通过 AI 辅助录入合同和租赁资料
- 采用更适合财报和审计的日级计量底座
- 由用户选择查询和输出粒度
- 支持自定义财年和自定义报告期间

建议 IT 团队以本文件作为需求增量清单阅读，再结合主文档开展详细设计。
