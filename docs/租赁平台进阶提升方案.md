# 租赁管理平台 — 商业进阶提升方案

> 版本：v1.0 ｜ 编制日期：2026-05-30 ｜ 性质：产品与商业策略文档
>
> 目的：在技术 MVP 已完成的基础上，明确"如何把作品变成可销售产品"的提升路径。本文档既是开发路线图，也是对外讲故事的依据。

---

## 0. 一句话定位

> **以 AI 录入为入口、以租赁全生命周期管理为日常、以 IFRS 16 合规计量为护城河的统一租赁平台。**
>
> 面向租了大量门店/设备的连锁企业：用 AI 把租约录入工时砍掉 80%，用关键日期预警避免错过续租/终止窗口，用事件驱动引擎保证每一笔数可追溯、过审计。

**定位演进**：IFRS 16 合规计量从"产品的全部"降级为"其中一个高价值模块"——但正是它让本平台比纯租赁管理工具更可信、更敢承诺过审计（详见 [主题四](#主题四从-ifrs-16-工具--统一租赁平台战略升级)）。

核心产品哲学：**让 AI 把脏活累活全包了，但把"能不能信、能不能过审计"的控制权牢牢留在系统和人手里。** 四个提升方向（正确性证明、范围判定、AI 入口、平台化升级）全部服务于这一句话。

---

## 1. 现状判断

### 1.1 价值内核不弱
IFRS 16 是强制准则，零售/连锁企业动辄上百到上千份门店租约，必须逐份算租赁负债、使用权资产、利息摊销、折旧，每月出分录、做披露、应对审计。这件事**痛、刚需、高频、合规驱动**——预算不需要被"说服"，这是最大的价值基础。

### 1.2 "可销售价值"目前偏弱
技术 MVP 完成度高，但停在"功能演示"层。弱点不在技术，而在**商业化要素**：

| 短板 | 说明 |
|------|------|
| **未证明算得准** | IFRS 16 工具的生死线是计量正确性。目前只有 1 份合同、1 个月结的验证。客户和审计师只会因为"敢不敢用它的数过审计"买单 |
| **价值未量化** | 描述的是功能清单，不是 ROI。答不上"省几个人天 / 降低多少审计调整 / 避免多少违规风险" |
| **集成是空的** | ERP/总账、AP、销售系统、SSO 全是"接口设计"，未落地。对企业客户这往往是硬门槛 |
| **AI 聊天是 stub** | Core Service 里是关键词匹配，非真实 LLM，演示会穿帮 |
| **单点工具非平台** | 缺多准则对比、合并抵销、敏感性分析与预测，客单价天花板低 |

> **结论：不是商业价值弱，而是商业价值"未被证明、未被量化、未被集成"。技术债不多，商业债很多。**

## 2. 四大进阶主题（本次讨论核心）

### 主题一：把"正确性"做成可信资产（决定生死）

**问题**：IFRS 16 工具的核心是算得对、敢过审计。当前验证样本严重不足。

**提升动作**：
1. 建一套 **20–50 份覆盖典型场景的回归测试用例**，每份都有第三方核对过的标准答案，做成**自动化对数报告**。场景至少覆盖：
   - 先付租金 / 后付租金
   - 续租选择权判断
   - 提前终止
   - 指数/费率调整（reassessment）
   - 变量租金（turnover rent）费用化
   - 减值触发
   - lease modification（范围/对价变化）
2. 产出一份 **《计量方法与准则映射白皮书》**，明确每个会计判断对应 IFRS 16 哪一条。
3. 把"对数报告"做成可一键运行、可导出的产物，作为售前材料。

**价值跃迁**：从"一个 demo"变成"敢承诺过审计的引擎"。**这是把弱价值变强价值最划算的一步。**

---

### 主题二：合同范围判定与分流计量（合规红线 + 差异化卖点）

**问题（关键洞察）**：**不是每一份合同都适用 IFRS 16 资本化，不需要算 ROU 和租赁负债。** 当前系统无条件资本化每一份合同——schema 无 `lease_scope` 等字段，`calculation.go` 无豁免分支。这会做出**错误的财务报表**，审计师第一眼就会否掉。

**应当区分的情形**：

| 情形 | 正确处理 | 当前系统处理 |
|------|---------|------------|
| **短期租赁**（租期≤12个月，无续租意图） | 可豁免，直线法费用化 | ❌ 强制资本化（高估资产负债） |
| **低价值资产**（单项新购价值低，如复印机、笔记本） | 可豁免，直线法费用化 | ❌ 强制资本化 |
| **非租赁合同**（无"已识别资产"或承租方无控制权，如纯服务/仓储/云） | 不属于 IFRS 16，不入表 | ❌ 误当租赁资本化 |
| **租赁+非租赁成分混合**（CAM、服务费等） | 拆分，非租赁成分不进负债 | ⚠️ 有约束声明，无强制拆分 |

**为什么这是高价值**：
- **合规风险**：无脑资本化短期租约和服务合同 = 报表错误 = 过不了审计。
- **专业壁垒**：判断"要不要上表"需结构化判定规则 + 留痕，正是 Excel 和廉价工具的软肋。
- **契合现有架构**：判定本质是从合同文本抽取关键事实（租期、资产类型、是否唯一识别资产、是否可替换），完美适配 AI 初判 + 置信度 + 人工复核的 Assist Mode 设计。

**最小可行落地（范围闸门，在进入计量引擎前拦一道）**：
1. **数据模型**：合同头新增字段
   - `lease_scope`：`in_scope` / `short_term_exempt` / `low_value_exempt` / `not_a_lease`
   - `exemption_reason`、`classified_by`、`classified_at`
2. **计量引擎分流**（`calculation.go`）：
   - `in_scope` → 走现有 ROU/负债计算
   - 豁免 / 非租赁 → **跳过资本化**，改走**直线法费用确认**，每月只生成租金费用分录，不生成负债滚动
3. **AI 初判**：解析时让 LLM 输出"是否租赁 + 是否可豁免 + 依据 + 置信度"，写入草稿，低置信度强制人工审
4. **报表披露**：分开列示已资本化租赁 vs 豁免/费用化租赁（IFRS 16 本身要求披露短期和低价值豁免的费用金额）——又一个客户买单理由

### 主题三：AI 助手作为文件上传主入口（差异化体验）

**问题（关键洞察）**：当前"不够勇敢"，不敢把 AI 助手做成文件上传入口。现有是重表单的 `/upload` 流 + 独立 `/ai-chat`（且为 stub）。

**为什么这是高价值差异化**：
- 市面 IFRS 16 工具几乎都是**重表单企业级 UI**，培训成本高。采购方真正痛点是"财务团队不愿用、录入太慢"。
- **Onboarding 摩擦 = 财务工具采纳率第一杀手**。"把合同丢给它、跟它说话就行"是极强的 demo 钩子与销售话术。
- 顺应 agentic copilot 趋势，认知和估值上高于"又一个租赁台账系统"。

**核心原则：勇敢在前门，严格在账本。** AI 当"超级录入员 + 助手"，但永远不当"会计主管"。

| 层 | 态度 | 做法 |
|----|------|------|
| **采集/入口层** | 🔥 大胆 | AI 助手即上传入口。对话里拖文件 → 自动 OCR+解析 → 当场出草稿 |
| **确认层** | ⚖️ 半结构化 | 对话里**不只回文字气泡**，渲染**结构化草稿卡片**（字段 + 置信度 + 原文高亮），逐条可改 |
| **过账层** | 🔒 保守 | 范围判定、计量、入正式台账、生成分录**仍走原有审批锁账**，AI 无过账权 |

**技术前置**：当前 AI 聊天是关键词 stub。要当入口，需先升级为**能调工具的真实 agent**：接收文件 → 调用上传 → 调用解析 → 创建草稿 → 把草稿卡片回渲到对话。这是从"聊天框"到"agentic intake"的升级。

**分两步走（不要一上来 all-in）**：
- **第一步（低风险高回报）**：保留现有 `/upload` 表单流不动，在 AI 助手中**新增**"拖文件 → 出草稿卡片 → 跳转确认"能力。两条路并存，用真实用户验证对话式入口接受度。
- **第二步（验证后）**：若数据证明对话入口更快更受欢迎，再升级为默认主入口，表单流降级为高级/批量模式。

### 主题四：从 IFRS 16 工具 → 统一租赁平台（战略升级）

**问题（最大战略洞察）**：不应让产品只做 IFRS 16 计算与管理。真正的市场是**租赁全生命周期管理**（行业称 Lease Administration），面向租了大量门店/设备的公司。

**为什么这是量级更大的市场**：

| 维度 | 只做 IFRS 16 | 做租赁全生命周期管理 |
|------|-------------|--------------------|
| 用户 | 财务部门 | 地产/租务/拓展/运营团队 + 财务 |
| 频次 | 每月一次（月结） | 每天都在用 |
| 痛点性质 | "算账麻烦" | "错过续租/终止窗口 = 真金白银损失" |
| 黏性/续费 | 中 | 高 |
| 市场验证 | — | Visual Lease、FinQuery、Nakisa、Occupier 等成熟公司已验证赛道 |

> **核心痛点举例**：一家有几百个门店租约的公司，最怕的不是算错负债，而是漏掉一个续租截止日 / break clause 通知期——一旦漏掉，可能被锁死再付数年租金，或丢掉黄金铺位。一次损失就是几十上百万。

**关键策略：IFRS 16 是"楔子"，租赁管理是"平台"**
- IFRS 16 是合规刚需，帮你敲开财务部门的门（预算好批）。
- 租赁管理让你变成天天被用的系统，黏住地产/运营团队。
- 市面多数租赁管理工具**会计能力很弱**；而你**会计引擎很强**。占位：**"既是最好用的租赁管理台账，又是敢过审计的 IFRS 16 引擎"**——别人占不到的位置。

**现有资产几乎可直接复用（是反帧 + 扩展，不是重写）**：

| 已有资产 | 在租赁管理平台中的角色 |
|---------|---------------------|
| 合同为核心的数据模型 | 天然就是租赁台账的中心 |
| 事件驱动（续租/终止/变更） | 关键日期与合同变更管理 |
| AI 文档解析 + 草稿层 | 价值更大（手工录几百份合同的条款/日期是最大痛点） |
| 多租户 + RBAC + 审计留痕 | 企业级台账标配 |

**租赁管理需补的核心能力（按价值排序，先做前三）**：
1. **集中合同库**：所有租约 + 附件/补充协议可搜可查，单一事实源。
2. **关键日期与提醒（最高价值）**：续租截止、break clause 通知期、租金 review 日、到期日、保险续保 → 自动预警。最能让客户"离不开"。
3. **条款/义务管理**：维修、CAM、保险、指数调整条款，结构化存储。
4. **组合分析**：总承诺租金、按区域/品牌/资产类型分布、到期分布图、每平米成本。
5. **设备租赁维度**：不只门店，还有车队/IT/机器设备（加 `asset_type` 维度）。

**用户分层与买家策略**：
- IFRS 16 买家 = **财务**；租赁管理买家 = **地产/运营**。两个买家、两套话术。
- 最优故事：**"一个平台，财务和运营都在用同一份合同数据"**——打通两个部门、提高客单价的杀手锏。

**需新增的数据模型（高层）**：
- `critical_dates`：合同关键日期表（类型、目标日、提前提醒天数、负责人、状态）。
- `lease_clauses` / `obligations`：条款义务表（类型、责任方、原文引用、结构化值）。
- `documents`：文档表（主合同/补充协议/side letter，版本、哈希、关联合同）。
- 合同头新增 `asset_type`（real_estate / vehicle / it_equipment / machinery / other），支撑设备租赁。
- 提醒/通知机制（到期前 N 天触发，可对接邮件/站内信）。

**要小心的两个坑**：
1. **范围爆炸**：租赁管理表面很大，别一次做全。先做"集中库 + 关键日期提醒 + AI 录入"三件。
2. **别丢掉 IFRS 16 差异化**：它是护城河和可信度来源，保持会计引擎的强度。

## 3. 其他配套提升（来自整体商业分析）

### 3.1 把价值翻译成钱（决定能不能谈下来）
- 做 **ROI 测算页**：输入合同数量，自动算"传统 Excel 工时 vs 本系统工时"、人力成本节省、审计风险敞口下降。
- 包装 2–3 个**可量化卖点**，例如"500 份租约月结从 5 人天 → 0.5 人天"、"消除手工公式错误导致的审计调整"。

### 3.2 打通一个真实集成（决定能不能落地）
- 不要全做。**选一个目标客户最常用的总账系统**（国内大概率用友/金蝶，外资 SAP/Oracle），真正跑通"分录导出 + 凭证号回写"一条链路。
- 一个能演示的真实集成，胜过六个 PPT 上的接口。

### 3.3 抬高天花板（决定客单价）
- **双准则/多准则对比**（IFRS 16 vs ASC 842 vs 本地准则）：外资和出海企业刚需。
- **敏感性与预测**：折现率/续租假设变动对负债的影响、未来现金流预测，把工具从"记账"升级到"决策支持"。

## 4. 商业模式建议

- **目标客户**：门店数 50+ 的连锁零售、餐饮、商业地产租户。
- **定价锚点**：按合同数量分层订阅，远低于客户养一个专职会计 + 审计返工的成本。
- **进入策略**：先做 1–2 家**标杆客户**真实跑一个完整财年，拿到"过审计"背书，再复制。

## 6. 总结

三个进阶主题——**正确性证明、范围判定、AI 入口**——指向同一个产品哲学：**让 AI 把脏活累活全包了，但把"能不能信、能不能过审计"的控制权牢牢留在系统和人手里。** 这正是区别于竞品的核心定位。

若资源有限只能做一件事：**先做主题一（带第三方背书的回归测试报告）**，这是把项目从"作品"变成"产品"性价比最高的一步。

## 附录 A：主题二"范围闸门"详细设计

目标：在合同进入 IFRS 16 计量引擎之前，加一道**范围判定闸门**，决定该合同是"资本化（算 ROU + 负债）"还是"豁免/非租赁（直线法费用化或不入表）"。

### A.1 数据模型变更

**合同头新增字段**（`contracts` 表）：

| 字段 | 类型 | 取值 / 说明 |
|------|------|------------|
| `lease_scope` | varchar | `in_scope`（资本化）/ `short_term_exempt`（短期豁免）/ `low_value_exempt`（低价值豁免）/ `not_a_lease`（非租赁，不入 IFRS 16） |
| `exemption_reason` | text | 豁免/排除依据说明（如"租期 10 个月且无续租意图"） |
| `scope_classified_by` | uuid | 范围判定确认人（FK users） |
| `scope_classified_at` | timestamptz | 确认时间 |
| `scope_source` | varchar | 判定来源：`ai_suggested` / `manual` / `policy_rule` |
| `scope_confidence` | numeric | AI 初判置信度（0–1），人工确认后可置空 |

**判定决策树（建议固化为政策规则，供 AI 与人工共用）**：
```
1. 该合同是否存在"已识别资产"且承租方控制其使用？
   否 → not_a_lease（纯服务/可替换资产/无控制权）
2. 租赁期（含合理确定行使的续租）是否 ≤ 12 个月且无购买选择权？
   是 → short_term_exempt
3. 标的资产单项全新价值是否低于低价值门槛（按公司政策，如 ≈ ¥35,000）？
   是 → low_value_exempt
4. 以上均否 → in_scope（进入完整 ROU/负债计量）
```

> 注：短期与低价值豁免按"资产类别"做政策选择，需记录公司政策版本号，纳入审计追溯。

### A.2 计量引擎分流（`calculation.go`）

在 `Calculate()` 入口处、任何 ROU/负债计算之前增加分流：

```go
// 伪代码示意，非最终实现
switch contract.LeaseScope {
case "in_scope":
    // 走现有逻辑：现值、ROU、利息摊销、折旧、负债滚动
    return CalculateCapitalized(contract, schedule, params)

case "short_term_exempt", "low_value_exempt":
    // 直线法费用确认：不生成负债/ROU，不做负债滚动
    // 每期租金费用 = 租赁期内固定付款总额 / 期数（直线法）
    return CalculateStraightLineExpense(contract, schedule)

case "not_a_lease":
    // 不属于 IFRS 16：不生成任何租赁会计结果
    // 仅作为台账记录存在（仍可用于租赁管理/关键日期等非会计模块）
    return SkipMeasurement(contract)
}
```

**月结分录差异**：
- `in_scope`：利息 + 折旧 +（付款冲减负债）——现有三类分录。
- 豁免类：**仅一笔租金费用分录**（借：租赁费用 / 贷：应付或银行），不生成负债滚动、不进负债滚动表。
- `not_a_lease`：不生成任何 IFRS 16 分录（其费用由其它科目/系统处理）。

> 实现要点：直线法需正确处理免租期（rent-free period）摊销和跨期分摊；豁免类仍要进入"租赁管理"模块（关键日期、合同库），只是不进会计计量。

### A.3 AI 初判（解析阶段）

合同解析时，让 LLM 在草稿中额外输出：
- `is_lease`（是否构成租赁）+ 依据
- `suggested_scope`（建议的 `lease_scope` 取值）+ 依据 + 置信度
- 触发条件：置信度低于阈值 → 强制人工审核（符合 Assist Mode 红线，AI 不得直接定性入库）。

### A.4 报表与披露

- 披露报表**分开列示**：已资本化租赁 vs 短期豁免费用 vs 低价值豁免费用（IFRS 16 本身要求披露短期与低价值豁免确认的费用金额）。
- 报表新增维度：按 `lease_scope` 汇总，便于审计核对"哪些合同被豁免、依据是什么"。

### A.5 影响面清单（落地时需触及）
- `db/init/01_init.sql`：`contracts` 表加字段 + 政策规则种子数据。
- `calculation.go`（handler 与计量核心）：入口分流 + 直线法费用函数。
- `monthly_closing.go`：豁免类只生成租金费用分录的分支。
- `reports.go`：披露报表按 scope 分列。
- AI Service `parse.py`：合同解析输出 `is_lease` / `suggested_scope` / 置信度。
- 前端合同页：范围判定字段展示 + 人工确认控件 + 草稿确认环节。

## 附录 B：主题一 回归测试用例清单（初版）

**校验口径**：初始租赁负债、初始 ROU、各期利息、各期折旧、期末负债、期末 ROU、月结分录金额；误差容忍 < ¥1（仅用于 rounding dust）。
